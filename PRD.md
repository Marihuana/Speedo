## 1. Project Overview

### 1.1 목적 (Purpose)
모터사이클 라이더의 스마트폰을 주행 계기판으로 활용하여, 주행 중 필요한 핵심 안전 및 성능 지표를 실시간으로 제공하고, 주행 기록을 저장하여 사후에 리뷰할 수 있도록 돕습니다.
- **실시간 주행 정보**: GPS 기반 속도, 센서 기반 차체 기울기 각도 및 최대 기울기 제공.
- **안전 지표**: 급제동 이벤트 감지 및 경고.
- **사후 리뷰**: 주행 경로, 속도 및 기울기 변화, 급제동 위치를 결합한 종합 주행 로그 제공.
- *※ TPMS(타이어 공기압) 기능은 센서 정합성 검증을 위해 이번 버전의 UI 노출 및 기능 범위에서 제외합니다.*

### 1.2 타겟 유저 및 사용 시나리오 (Target User & Scenario)
- **타겟 유저**: 스마트폰을 모터사이클에 거치하여 사용하는 라이더.
- **사용 환경**: 야외 주행 환경 (화면 상시 시인 필요, GPS 수신 필수).
- **대표 시나리오**:
  1. 라이더가 주행 시작 전 앱을 켜고 '기록 시작'을 누릅니다.
  2. 주행 중 계기판(속도, 기울기 게이지)을 보며 주행하고, 급제동 시 즉각적인 피드백을 받습니다.
  3. 주행이 끝나면 바이크를 세우고 시동을 끕니다. 장시간 정차가 감지되면 앱이 스스로 주행 기록을 안전하게 저장하고 종료를 알립니다.
  4. 복귀 후, 지도 위에 그려진 주행 경로와 구간별 속도/기울기를 분석합니다.

### 1.3 수익 모델
- **광고 탑재**: 기본 서비스는 무료로 제공되며, AdMob 배너 및 전면 광고가 포함됩니다.
- **구독형 모델 (Subscription)**: 앱 내 모든 광고를 제거하는 구독 요금제를 제공합니다. 최초 구독 시 **14일 무료 체험(14-day Free Trial)** 혜택이 제공됩니다.
  - **월 구독 (Monthly Subscription)**: 1,500원 ($0.99)
  - **연 구독 (Annual Subscription)**: 12,000원 ($8.99)
- **일회성/소비성 후원 모델**: 설정 화면 하단에 후원 버튼을 제공하며, 사용자가 반복 결제할 수 있는 일회성(소비성) 인앱 후원 상품을 제공합니다. (상세 후원 금액 및 결제 유도 문구는 TBD)

## 2. Core User Flows & Features

### 2.1 화면 및 비즈니스 흐름 (Screen & Navigation Flow)
- **앱 시작 (Splash)**:
  - 필수 권한인 **위치 권한**을 요청합니다. 위치 권한이 거부될 경우, 안내 팝업을 노출하고 앱을 종료합니다.
  - 선택 권한인 **알림 권한**을 함께 요청하되, 거부되더라도 앱 진입을 막지 않습니다.
  - 권한 결정 완료 후 2초 동안 스플래시 화면을 유지한 뒤 메인 화면으로 이동합니다.
  - **[F-25] 최초 실행 안내 화면**: 최초 실행 시 메인 화면 진입 직전에 안전 가이드 및 스마트폰 거치 안내를 전면 형태로 노출합니다. (디자인 참고: [Figma Node 80-41](https://www.figma.com/design/HwtqT6Wp2w5qcpZMBL2Xuv/Every-Bari?node-id=80-41&m=dev), [Figma Node 80-291](https://www.figma.com/design/HwtqT6Wp2w5qcpZMBL2Xuv/Every-Bari?node-id=80-291&m=dev))
- **메인 화면**: 좌우 스와이프로 전환 가능한 3개 탭으로 구성됩니다.
  - `Monitor(대시보드)` / `Records(주행 목록)` / `Settings(설정)`
  - **[F-26] 가로모드 UI 개선**:
    - **Monitor(대시보드) 가로모드**: 스피드미터 높이를 최대화하여 시인성을 확보하고 우측에 게이지 배치. (디자인 참고: [Figma Node 80-333](https://www.figma.com/design/HwtqT6Wp2w5qcpZMBL2Xuv/Every-Bari?node-id=80-333&m=dev))
    - **Records(주행 목록/상세) 가로모드**: 상세 경로 지도와 텔레메트리 정보를 좌우 가로 분할 형태로 노출. (디자인 참고: [Figma Node 80-447](https://www.figma.com/design/HwtqT6Wp2w5qcpZMBL2Xuv/Every-Bari?node-id=80-447&m=dev))
- **주행 로그 진입**: 주행 목록에서 특정 항목을 탭하면 해당 주행의 상세 로그 화면으로 이동합니다.
- **앱 종료 정책**: 메인 화면에서 뒤로가기 버튼을 2초 이내에 2회 연속으로 누를 때만 앱을 종료하며, 1회 입력 시에는 안내 토스트를 노출합니다.

### 2.2 기능 요구사항 명세 (Features & Fixes)

| # | 기능/버그명 | 비즈니스 목적 | 동작 조건 (Trigger) | 기대 동작 (To-Be) | 비고 |

| **F-03a**| 기울기 게이지 스무딩 | 눈의 피로를 줄이고 직관적인 시각 피드백 제공 | 기울기 데이터가 0.1초(100ms) 주기로 갱신될 때 | UI 레이어에서 각도 변화가 급격히 끊어지지 않도록 부드러운 애니메이션(보간)을 적용하여 게이지와 텍스트를 표시함. (지연 시간 최대 250ms 이하 유지) | |
| **F-03b**| 물리적 뱅킹각 가드 (Sensor Guard) | 정차 시 폰 조작이나 극저속 흔들림으로 인한 뱅킹각 데이터 오염 방지 | 주행 기록 중 실시간 속도 및 뱅킹각 데이터 수신 시 | 정상 원선회 모델을 참고한 자이로 요율(Yaw Rate) 및 속도 기반 예측 모델을 통해 데이터의 신뢰도(LeanConfidence)를 판정하고, 물리적으로 불가능한 노이즈를 식별하고 보정함. 또한 실시간 현재 뱅킹각은 필터링 없이 원시 데이터를 표시하고, 실시간 최대 뱅킹각 및 사후 로그 UI(최대 뱅킹각 통계, 지도 경로 시각화 등)는 VALID 데이터 기준으로만 반영함. (세부 사양은 4.1절 참조) | **[구현 완료]** (LeanPhysicsGuard 및 Peak 갱신 연동 완료) |

## 3. Business Data Requirements

### 3.1 비즈니스 데이터 모델 및 제약조건

기획서 상에서 관리하는 핵심 도메인 데이터 구조와 비즈니스적 제약 조건입니다.

#### 1) 주행 기록 요약 (Ride Summary)
*   **주행 식별자**: 각 기록을 구분하는 고유 ID
*   **주행 제목**: 주행을 대표하는 텍스트 (기본값은 기록 시작 날짜/시간 포맷이며, 사용자가 공백이 아닌 텍스트로 자유롭게 수정 가능)
*   **시간 정보**: 주행 시작 시각 및 종료 시각
*   **누적 주행 거리**: km 단위로 소수점 첫째 자리까지 표시
*   **최대 뱅킹각**: 주행 중 발생한 최대 기울기 각도 (절대값 기준, 도 단위 정수 표시)
*   **최고 속도**: 주행 중 기록된 가장 높은 속도 (km/h 단위 정수 표시)
*   **총 주행 시간**: 주행 시작부터 종료까지의 시간 (시:분:초 포맷 표시)

#### 2) 실시간 텔레메트리 데이터 (Telemetry Log)
*   **시간 정보**: 데이터가 측정된 시각 (밀리초 단위)
*   **실시간 속도**: 순간 속도 (km/h)
*   **실시간 뱅킹각**: 순간 기울기 각도 (좌/우 방향을 구분하기 위해 부호가 포함된 값으로 관리)
*   **급제동 이벤트**: 해당 시점의 제동 상태 등급 (`NONE`, `LIGHT`, `MODERATE`, `HARD`) 및 제동 강도 수치
*   **위치 좌표**: 위도 및 경도 (GPS 수신 성공 시에만 존재하며, 음영 구간이나 고주기 센서 기록 시점에는 존재하지 않을 수 있음)

#### 3) 정비 기록 (Maintenance Record)
*   **정비 식별자 (ID)**: 각 정비 기록을 구분하는 고유 ID (UUID)
*   **정비 등록 시간 (Created At)**: 정비 기록이 생성된 시스템 시간 타임스탬프 (ISO 8601 표준 포맷 `YYYY-MM-DDTHH:mm:ssZ` 등으로 기록하여 시간대 일치시킴)
*   **정비 날짜 (Date)**: 실제 정비가 수행된 날짜 (미래 날짜 선택 불가, `YYYY-MM-DD` 포맷)
*   **정비 항목 (Maintenance Type)**: 프리셋 Enum 값 또는 직접 입력한 문자열
*   **정비 시 주행거리 (Odometer)**: km 단위 정수. 필수 입력 사항이며 0 이상의 값이어야 함. (바이크 자체의 계기판 주행거리 기준 입력, 앱의 주행 기록 누적 거리와 연동하지 않음)
*   **정비 비용 (Cost)**: 원(KRW) 또는 설정된 통화 단위 of 정수. 선택 입력 사항이며 0 이상의 값이어야 함. 기기의 지역 설정(Locale)을 감지하여 통화 기호(예: ₩, $)를 자동 표시.
*   **정비처 (Shop)**: 최대 50자 이내의 문자열 (선택)
*   **메모 (Notes)**: 최대 1,000자 이내의 자유 텍스트 (선택)

## 4. Exception Handling & Edge Cases

### 4.1 시스템 및 외부 입력 예외
- **GPS 수신 불가 (터널, 지하 등 음영 구간)**:
  - 실시간 속도는 0으로 처리하지 않고 직전 수신 속도를 유지하거나 공백 처리합니다.
  - 음영 구간의 이동 경로는 지도 상에서 누락되지 않도록 **진입 직전 위치와 진출 직후 위치를 기반으로 시간 비례 보간(F-13d)**하여 경로를 복원하고 속도를 재계산합니다.
- **GPS 신호 튐 방지 가드 (GPS Signal Guard)**:
  - **물리적 한계 속도 임계치 적용**:
    - 초고성능 튜닝 바이크 및 서킷 주행 성능을 감안하여, 절대 속도 한계 임계값을 **350km/h**로 설정합니다. 수신된 GPS 속도가 350km/h를 초과할 경우, 이는 센서 오류 또는 신호 튐으로 판단하여 즉시 폐기합니다.
  - **궤적 일관성 필터 (Trajectory Consistency Filter)**:
    - 직전 수신된 신뢰할 수 있는 주행 경로 벡터(진행 방향 및 속도)와 현재 수신된 GPS 좌표를 비교 분석합니다.
    - 새로 수신된 좌표가 물리적으로 불가능한 방향 전환(예: 진행 방향의 급격한 역전 등)이나 비현실적인 도약 거리(직전 속도 대비 허용 오차 범위를 크게 벗어난 거리)를 보일 경우, 이를 이상치(Outlier)로 판정합니다.
    - 이상치로 판정된 좌표는 기록에서 아예 누락된 것(미수신/Drop)으로 취급하여 영구 폐기합니다.
    - **이상치 구간 선형 보간**:
      - 이상치 판정으로 인해 좌표가 폐기된 후, 다시 신뢰도가 확보된 정상 좌표가 수신되면 이상치 발생 직전의 마지막 신뢰 좌표와 해당 정상 좌표 사이의 구간을 시간 비례 선형 보간(F-13d)으로 연결하여 경로 튐 현상을 원천 차단합니다.
- **기울기 센서 측정 불능**:
  - 센서 값 수신이 일시적으로 중단되거나 오류 값이 들어올 경우, 화면에 뱅킹각을 0°로 고정 표시하여 비정상적인 게이지 흔들림을 방지합니다.
- **물리적으로 불가능한 센서 데이터 필터링 (자이로 요율 기반 예측 뱅킹각 검증 및 가드)**:
  - 주행 종료를 잊었거나, 정차 및 극저속 상태에서 스마트폰을 조작/탈착할 때 발생하는 뱅킹각 오염을 방지하고 물리적으로 타당한 데이터만 기록하기 위해, 오직 **자이로 요율(Yaw Rate) 기반 예측 뱅킹각 검증 모델**을 적용합니다. (바이크 기종별 제한 및 `MotorcycleProfile` 테이블을 통한 강제 임계값 제한 정책은 적용하지 않습니다.)
  - **동작 로직 및 물리식 검증**:
    - 실시간 속도 $v$ (m/s)와 자이로 요율(Yaw Rate) $\omega$ (rad/s)를 이용하여 예측 뱅킹각 $\theta_{expected}$를 다음과 같이 계산합니다:
      $$\theta_{expected} = \arctan\left(\frac{v \cdot \omega}{g}\right)$$
      *(단, $g$는 중력가속도 $9.81\text{ m/s}^2$)*
    - 실제 센서로 측정한 롤(Roll)각과 예측 뱅킹각 $\theta_{expected}$의 오차를 비교합니다.
    - 비교 결과, 오차가 속도 구간별 오차 허용치(Tolerance)를 초과하는 경우 이를 폰 조작 등 외부 요인에 의한 노이즈(`OUTLIER_NOISE`)로 간주하고, 측정값을 예측 뱅킹각 $\theta_{expected}$로 제한(Clipping)하여 보정 저장합니다.
  - **속도 구간별 허용 오차 (Tolerance)**:
    - $5\text{ km/h} \le v < 15\text{ km/h}$: $\pm 20^\circ$
    - $15\text{ km/h} \le v < 30\text{ km/h}$: $\pm 15^\circ$
    - $30\text{ km/h}$ 이상: $\pm 12^\circ$
  - **극저속 구간 예외 처리 ($v < 5\text{ km/h}$)**:
    - 극저속 구간에서는 원시 데이터를 무조건 0°로 평탄화하지 않고, 데이터 신뢰도(LeanConfidence)를 `LOW_SPEED_UNRELIABLE`로 마킹합니다.
    - 정차 상태에서 사이드 스탠드 거치나 발로 차체를 버티는 등의 실제 기울기를 자연스럽게 반영하기 위해, 실제 측정된 각도를 기반으로 다음과 같이 처리합니다.
      - **15° 이하의 뱅킹각**: 실제 측정된 원시 각도 그대로 실시간 대시보드 UI에 표시하고 주행 상세 로그(지도 경로 포함)에 정상 기록 및 렌더링합니다.
      - **15°를 초과하는 뱅킹각**: 정차 중 폰 조작 노이즈로 간주하여 최대 **15°로 제한(Clipping)**하여 실시간 대시보드에 표시하고 상세 로그에 저장 및 렌더링합니다.
  - **LeanConfidence 등급에 따른 구체적인 UI 노출 및 통계 반영 정책**:
    - **실시간 대시보드 UI 정책**:
      - **실시간 현재 뱅킹각 (Current Lean Angle)**: 신뢰도 등급(`LOW_SPEED_UNRELIABLE`, `OUTLIER_NOISE` 등) 및 속도와 관계없이 센서가 측정하는 원시 각도(Raw Roll) 그대로 실시간 게이지와 수치 텍스트에 투명하게 표시합니다. (정차 중 폰 조작 시 화면이 즉각 반응하도록 하여 센서 동작 직관성을 확보합니다.)
      - **실시간 좌우 최대 뱅킹각 (L/R Peak Lean Angle)**: 대시보드 화면에 표시되는 주행 중 최대 뱅킹각 기록은 오직 신뢰도가 `VALID` 상태인 데이터에 의해서만 갱신(Update)되도록 제한합니다. (예: 정차 중 폰을 45도로 만져 '현재 뱅킹각'이 45도로 표시되더라도, '최대 뱅킹각' 카드는 갱신되지 않고 이전의 정상 주행 최대치를 유지합니다.)
    - **주행 상세 로그 및 지도 경로 (F-13, F-13a) UI 정책**:
      - **최종 최대 뱅킹각 (Ride Summary)**: Ride Summary의 최종 최대 뱅킹각 통계는 기존과 동일하게 오직 `VALID` 상태의 데이터만을 기준으로 산출합니다.
      - **지도 상의 경로 및 상세 데이터 뷰**: LeanConfidence 등급에 따라 지도 상의 경로 색상 시각화 및 상세 데이터 뷰에 다음과 같이 다르게 렌더링합니다:
        - **VALID 상태**: 측정된 각도 그대로 지도 경로 색상 및 상세 데이터 뷰에 렌더링함.
        - **LOW_SPEED_UNRELIABLE 상태**: 정차 시 스탠드 거치/발 지탱 각도를 렌더링하기 위해, **최대 15도 제한(Clipping)이 적용된 보정된 각도(15도 이하 원시값, 15도 초과 시 15도)를 기준으로 지도 상의 경로 색상 시각화 및 상세 데이터 뷰에 렌더링함.**
        - **OUTLIER_NOISE 상태**: 데이터 포인트를 **0°**로 평탄화(초록색/수평 처리)하여 렌더링함으로써 경로 오염을 방지함.
- **주행 자동 종료 시 버퍼 데이터 트리밍(Trimming) 및 Ride Summary 데이터 재연산 정책**:
  - 사용자가 주행 종료를 누르지 않고 정차하여 설정된 임계 시간(T분, 예: 5분) 동안 저속(7km/h 미만) 상태를 유지하여 자동 종료가 트리거되는 경우를 대상으로 합니다.
  - 시스템은 전체 수집된 텔레메트리 로그 중 자동 종료 트리거 시점으로부터 역산하여 T분 동안의 데이터를 삭제 처리(Trimming)합니다.
  - 최종 저장되는 Ride Summary의 `총 주행 시간` 및 Google Map 상의 `주행 경로 궤적`은 정차 시점(자동 종료 트리거 T분 전)을 기준으로 잘라내어 저장됩니다.
  - **정합성 보장 재연산**: Trimming이 수행되면, 잘려 나간 T분 구간을 제외한 나머지 이전 구간의 텔레메트리 데이터 전체를 재탐색하여 Ride Summary의 `최대 뱅킹각`, `최고 속도`, `누적 주행 거리`를 재연산(Recalculate)한 후 DB에 최종 저장합니다. (트리밍 대상 저속 구간에서 발생한 폰 조작 등으로 인한 뱅킹각 오류 및 GPS 튐으로 인한 최고 속도 오차가 최종 통계치에 반영되지 않도록 원천 차단합니다.)

### 4.2 데이터 공백 및 초기 상태 (Empty State)
- **저장된 주행 기록이 없는 경우**:
  - 주행 목록 화면은 단순히 비워두지 않고, *"저장된 주행 기록이 없습니다. 새로운 주행을 시작해 보세요!"* (영문: *"No rides recorded. Start a new ride!"*)와 같은 명확한 안내 문구를 제공합니다.
- **경로 데이터가 너무 짧은 주행 기록 (예: 실수로 켜고 바로 끈 경우)**:
  - 총 주행 거리가 10m 미만이거나 주행 시간이 5초 미만인 주행은 기록 종료 시 자동으로 저장하지 않고 삭제합니다.

### 4.3 입력 유효성 검증 실패 (Validation Failure)
- **주행 제목 수정 시**:
  - 공백 문자만 입력하거나 빈 칸으로 수정하려고 할 경우, '저장' 버튼을 비활성화하고 기존 제목을 유지하도록 유도합니다.

### 4.4 사용자 환경 예외 (권한 거부 등)
- **주행 중 위치 권한이 취소되는 경우**:
  - 포그라운드 서비스 동작 중 시스템 설정에서 사용자가 위치 권한을 강제로 취소할 경우, 안전하게 기록을 자동 종료 및 저장하고 상태 알림을 통해 *"위치 권한이 해제되어 주행 기록이 종료되었습니다."*라고 안내합니다.
- **오버레이 권한 거부 시**:
  - 설정에서 플로팅 오버레이 기능을 켤 때 `다른 앱 위에 표시` 권한이 없다면, 권한 허용이 필요한 이유를 알리는 다일로그를 띄운 후 시스템 설정 화면으로 바로 이동시킵니다.

### 4.5 정비 이력 관련 예외 처리 및 엣지 케이스
- **미래 날짜 입력 시도**:
  - 정비 날짜 선택 피커(Calendar Picker)에서 오늘 이후의 날짜는 선택할 수 없도록 비활성화 처리합니다.
- **정비 이력 목록의 정렬 일관성**:
  - 목록 조회 시 정비 날짜(Date) 최신순으로 정렬하되, 날짜가 동일한 이력이 있을 경우 정비 등록 시간(Created At) 최신순으로 정렬하여 일관된 뷰를 유지합니다.
- **백업 파일 손상 또는 유효하지 않은 포맷**:
  - 복구(Import) 진행 시, 선택한 파일이 유효한 JSON 포맷이 아니거나 다음 메타데이터 스펙을 포함하지 않는 경우(타 OS에서 백업된 파일 포함), 복구를 즉각 중단하고 에러 팝업을 노출하여 사용자에게 알립니다.
    * **필수 메타데이터 스키마**: `{ "metadata": { "app": "Every Bari", "data_type": "maintenance_backup", "version": 1 } }`
  - 복구된 데이터 내의 시간 정보는 시스템의 ISO 8601 파서를 사용하여 타 OS 기기에서 생성된 시간 오프셋도 현재 기기의 로컬 시간대로 오차 없이 변환하여 반영합니다.

### 4.6 최초 안내 노출 및 가로모드 전환 시 엣지 케이스 예외 처리
- **최초 실행 상태 저장(Jetpack DataStore) 동기화 오류 및 화면 깜빡임(Jank) 방지**:
  - 앱 최초 설치 시 디폴트 값은 `is_first_launch = true`로 설정하며, 안내 문구 화면에서 확인 버튼을 명확하게 터치하여 상태가 업데이트(`false`)되기 전까지는 강제 종료, 백그라운드 전환 후 복귀 등 어떠한 시나리오에서도 매 진입 시 안전 안내 화면을 계속 노출함.
  - **스토리지 예외 대책**: Jetpack DataStore의 흐름(Flow)을 사용하여 비동기 원자적 쓰기를 보장함. 만약 디바이스 디스크 Full 등 예외적인 IO 에러 발생 시 즉시 Firebase Crashlytics에 비치명적 에러(Non-fatal)를 기록하고 메모리 내 임시 캐시(`isFirstLaunchMemoryCache`)로 대체 작동하여 당해 세션의 동작을 보장함. 연속적으로 쓰기 실패 시 임시 알림(토스트 또는 다이얼로그)을 통해 기기 저장 공간을 확보하도록 사용자에게 안내함.
  - **트랜지션 Jank 및 깜빡임 방지**: Splash Screen API의 `setKeepOnCondition`을 통해 `is_first_launch` 플래그 리딩이 완료될 때까지 스플래시 화면을 유지함. 읽기가 완료되면 NavGraph의 시작 목적지(Start Destination)를 `GuideFragment/Activity` 또는 `MainActivity`로 비동기 조건 분기 처리하여 라우팅함으로써 메인 화면이 순간적으로 보였다가 안내 화면으로 덮이는 Visual Leak 현상을 원천 방지함.
- **주행 기록 중 가로/세로 화면 회전 시 센서 및 데이터 연속성 유지**:
  - 주행 기록이 진행 중인 상태(포그라운드 서비스 활성화)에서 사용자가 기기를 가로 또는 세로로 회전할 때, 기록 세션(GPS 수집, 타이머, 주행 상태 등)이 초기화되거나 끊기지 않아야 함. Activity 재성장(Recreation) 시 ViewModel의 상태가 안전하게 복원되거나, 구성 변경(Configuration Change) 시 주행 제어 Service와의 바인딩이 유실되지 않도록 설계함.
  - **바인딩 transient state 리스크 방지**: Activity의 파괴와 재성장 과정에서 `ServiceConnection`이 해제되고 재결합하는 짧은 시간 동안 발생하는 실시간 센서 데이터 유실을 막기 위해, Service 단에서 텔레메트리 큐를 내부적으로 버퍼링(수집 누락 0ms 보장)하고 Activity가 재바인딩되는 즉시 최근 큐 데이터를 일괄 갱신(Pull 모델)받는 동기화 프로토콜을 구현함.
  - **자이로 센서 영점 보정 오프셋 유지 및 축 재매핑**: 가로모드로 전환 시 센서 영점 보정 오프셋(Offset) 값은 세로모드에서 설정한 값과 정확히 연동하여 동일하게 유지되어야 하며, 회전 시 새로운 영점 보정이 임의로 트리거되지 않아야 함. 기기의 물리적 회전에 따른 센서 좌표계 전환(예: 세로 Roll 축의 하드웨어 Y축에서 가로 Roll 축의 하드웨어 X축으로 변경)을 고려하여 `SensorManager.remapCoordinateSystem()`을 사용해 화면 방향(Rotation 0/90/180/270도)에 맞춰 3차원 회전 공간 상의 축 매핑을 보정한 뒤 물리적 Roll 각도와 영점 보정을 정합시킴.
- **가로모드 레이아웃의 해상도 및 비율 대응**:
  - 태블릿, 폴더블, 가로세로 비율이 독특한 디바이스(예: 21:9 비율 등) 등 가로 해상도가 넓은 환경에서도 스피드미터 텍스트 및 게이지 레이아웃이 화면 밖으로 잘리거나 겹치지 않도록 유동적인 뷰 크기(Dynamic Constraint/Aspect Ratio)를 가짐.
  - **분할 비율 및 Autosizing 범위 명시**: 가로모드 2열 분할 레이아웃(Monitor 탭) 적용 시 디바이스 해상도에 맞게 좌우 열 너비 비율을 `5:5` 또는 `6:4` 고정 비율로 유지하고, 최대 너비 제한(Max Width Constraint)을 설정하여 광활한 스크린(예: 태블릿)에서 뷰가 멀리 벌어지는 현상을 방지함. 속도계 텍스트 짤림 방지를 위한 Autosizing 설정 시 텍스트 크기의 최솟값을 `48sp`, 최댓값을 `120sp`로 엄격히 제한하여 비정상적으로 폰트 크기가 변하는 불균형을 방지함. 극단적인 해상도의 폴더블폰 커버 화면 등에서는 1열 스크롤 레이아웃으로 자동 폴백됨.

## 5. 인수 조건 (Acceptance Criteria)

### 5.1 기능별 검증 시나리오

#### AC-01: 위치 권한 거부 시 앱 진입 차단 (스플래시 화면)
*   **Given**: 사용자가 앱을 처음 설치하고 실행한 상태에서
*   **When**: 스플래시 화면에서 필수 권한인 '위치 권한' 요청을 거부할 때
*   **Then**: "위치 권한이 허용되지 않아 앱을 시작할 수 없습니다..." 안내 다이얼로그가 표시되고, 사용자가 확인을 누르면 앱이 정상 종료된다.

#### AC-02: 주행 기록 종료 시 광고 노출 및 화면 전환
*   **Given**: 사용자가 구독을 보유하지 않은 무료 사용자(구독 미보유 무료 사용자)이고, 주행 기록을 진행 중인 상태에서
*   **When**: 대시보드에서 '기록 종료' 버튼을 탭하고 저장 확인을 완료할 때
*   **Then**: 앱은 사용자의 구독 활성화 여부를 검증한다. 구독 미보유 상태임이 확인되면, 주행 기록을 즉시 중단 및 저장한 후 화면에 전면 광고를 1회 노출한다. 광고를 닫으면 메인 대시보드로 복귀한다.

#### AC-03: 주행 자동 종료 기능 동작 및 알림
*   **Given**: 설정에서 자동 종료 감지 시간이 '5분'으로 활성화되어 있고 주행 기록이 진행 중인 상태에서
*   **When**: 사용자가 오토바이를 멈추거나 7km/h 미만의 속도로 5분 이상 머무를 때
*   **Then**: 별도의 팝업 확인 없이 앱이 주행 기록을 즉시 안전하게 종료 및 저장하고, 상태 표시줄 알림(또는 포그라운드 알림)으로 "주행 종료가 감지되어 주행 기록을 종료했습니다." 메시지를 발송한다.

#### AC-04: 플로팅 오버레이 위젯 탭 시 앱 복귀
*   **Given**: 주행 기록 중 앱이 백그라운드로 전환되어 화면에 플로팅 오버레이 위젯이 활성화되어 있는 상태에서
*   **When**: 사용자가 플로팅 오버레이 위젯 영역을 탭(클릭)할 때
*   **Then**: 백그라운드에 있던 에브리바리 (Every Bari) 앱이 활성화(포그라운드 전환)되며 대시보드 화면이 전면에 나타난다.

#### AC-05: 정비 데이터 복구 기능 검증
*   **Given**: 사용자가 설정 화면에서 '정비 데이터 불러오기(복구)'를 선택하고 유효한 백업 파일을 지정한 상태에서
*   **When**: `maintenance_restore_confirm` 다이얼로그에서 '진행'을 선택할 때
*   **Then**: 백업 파일의 데이터가 로컬 DB에 정상적으로 덮어씌워지며 복구 완료 토스트가 표시되고, `Garage` 탭 진입 시 복구된 정비 목록이 정상적으로 조회된다.

#### AC-06: 최초 실행 시 안전 안내 노출 및 다시 보지 않기 동작
*   **Given**: 사용자가 앱을 처음 설치하고 위치 권한을 승인한 후 메인 화면 진입 전 상태에서
*   **When**: Splash Screen이 노출되고 메인 화면으로 이동하기 직전
*   **Then**: 로컬 저장소(DataStore)에서 `is_first_launch` 플래그를 읽어와 Splash가 종료되는 시점에 즉각 전면 안내 화면이 렌더링되며, 메인 화면의 대시보드 레이아웃이 찰나에 노출(Visual Leak)되는 일이 없어야 한다. 만약 저장소 IO 에러가 발생한 경우에는 임시 메모리 캐시 플래그에 의해 최초 1회만 노출된다.
*   **And**: 사용자가 "확인 (다시 보지 않기)" 버튼을 클릭하여 로컬 저장소 쓰기 작업(DataStore Write)이 완료되면, 안내 화면이 닫히고 메인 화면(대시보드)으로 진입한다. 이후 앱을 완전히 강제 종료했다가 재실행할 때도 플래그가 보존되어 해당 안내 화면이 노출되지 않고 바로 메인 화면으로 진입한다.

#### AC-07: 주행 기록 중 가로/세로 화면 전환 시 데이터 및 상태 보존
*   **Given**: 사용자가 세로모드에서 주행 기록을 시작하여 기록이 진행 중(예: 주행 시간 1분 경과, 누적 거리 0.5km)이고, 뱅킹각 영점 보정을 완료한 상태에서
*   **When**: 기기를 가로로 회전하여 가로모드로 전환할 때 (Activity 파괴 및 재성장 발생 시)
*   **Then**: 주행 기록을 담당하는 포그라운드 서비스의 센서/GPS 수집이 끊김 없이 유지되어 기록 데이터가 유실(유실 시간 0ms)되지 않고 연속적으로 누적된다.
*   **And**: Activity가 다시 생성되어 Service에 재바인딩되는 즉시 최신 텔레메트리 큐 데이터가 동기화되어 가로모드 UI로 부드럽게 레이아웃이 갱신(갱신 지연 시간 250ms 이하)된다.
*   **And**: 기록 중인 누적 주행 시간, 누적 거리 및 세로모드에서 적용했던 뱅킹각 영점 보정 오프셋 값(회전 축 매핑 보정이 적용된 상태)은 회전 전후 뱅킹각 오차 ±1° 이내의 정밀도로 유지되며 계속 연산된다.

- **[v1.2] 3. 랭킹 점수 시스템 (Ranking Score System) 세부 기획 및 정책 수립** (F-22)
  - **수식 초안 (가중 결합 모델)**:
    $$Score_{total} = \text{Round}\left( w_d \cdot D_{valid} + Score_{lean} + w_c \cdot C_{corner, valid} \right)$$
    - $D_{valid}$: `GpsSignalGuard`를 통과한 신뢰할 수 있는 누적 주행 거리 (km)
    - $Score_{lean}$: `LeanPhysicsGuard`를 통과하고 `LeanConfidence`가 `VALID` 판정을 받은 최대 뱅킹각 기반 기본 점수
      - $Score_{lean} = w_l \cdot \theta_{max, valid} \cdot \left(1 + \frac{\theta_{max, valid}}{45}\right)$ ($\theta_{max, valid}$는 절대값)
    - $C_{corner, valid}$: 신뢰할 수 있는 독립적 코너링 이벤트 횟수
      - **코너링 판정 기준**: 주행 속도가 $20\text{ km/h}$ 이상인 상태에서, 뱅킹각이 좌/우 $15^\circ$ 이상으로 최소 $1.5$초 이상 유지되었다가 수평($5^\circ$ 이하)으로 복귀한 경우를 1회 인정함.
    - 가중치 계수 최종안:
      - $w_d = 5$ (거리 가중치: 1km당 5점)
      - $w_l = 5$ (뱅킹 기본 가중치)
      - $w_c = 5$ (코너당 가중치)
  - **밸런싱 시나리오 예시 (기획 밸런스 검증)**:
    - **장거리 투어링 라이더 (속초 왕복 400km 주행)**:
      - 거리 점수: $5 \times 400 = 2,000$ 점
      - 뱅킹 점수: 최대 뱅킹각 $12^\circ$ 기준, 약 $80$ 점 ($Score_{lean} = 5 \times 12 \times (1 + 12/45) \approx 76$ 점)
      - 코너링 점수: 거의 없음 ($0$ 점)
      - **최종 점수: 약 2,080점**
    - **스포츠/와인딩 라이더 (유명산 코스 20km 주행, 최대 뱅킹각 35도, 코너링 40회 달성)**:
      - 거리 점수: $5 \times 20 = 100$ 점
      - 뱅킹각 점수: 최대 뱅킹각 $35^\circ$ 기준, 약 $311$ 점 ($Score_{lean} = 5 \times 35 \times (1 + 35/45) \approx 311.1$ 점)
      - 코너링 점수: $5 \times 40 = 200$ 점
      - **최종 점수: 약 611점**
      - *※ 균형 효과: 거리 가중치를 낮춤으로써(10 $\rightarrow$ 5), 상대적으로 짧은 와인딩 주행을 하더라도 깊은 코너링과 다수의 코너 카운트를 통해 고효율로 점수를 획득할 수 있는 스포츠 라이딩 특화 밸런스를 달성함.*
  - **데이터 신뢰 검증 및 예외 처리 방안**:
    - `GpsSignalGuard`에서 이상치로 판단되어 폐기 및 보간된 거리 데이터는 점수 계산에서 제외함.
    - 정차 중 스마트폰을 만져서 발생한 뱅킹각(즉, `LeanConfidence`가 `OUTLIER_NOISE` 또는 `LOW_SPEED_UNRELIABLE`인 데이터)은 랭킹 점수 및 코너 카운트 연산에서 제외함.
    - 어뷰징 방지: 평균 속도 $10\text{ km/h}$ 미만이거나 총 주행 시간이 $1\text{분}$ 미만인 경우 랭킹 점수를 부여하지 않음.
  - **시즌제 운영 정책**:
    - **본 시즌 (Main Season)**: 매년 3월 1일 ~ 11월 30일 (9개월간). 랭킹 점수를 누적하여 유저 간 경쟁을 활성화하고, 시즌 종료 시 최종 등급 부여 및 보상을 제공함.
    - **프리시즌 / 오프 시즌 (Pre/Off-season)**: 매년 12월 1일 ~ 익년 2월 28일 (3개월간). 겨울철 빙판길 안전사고 방지 및 라이더 보호를 위해 랭킹 점수 경쟁을 동결하며, 기록 누적이 없는 프리시즌 특별 연습 모드를 운영함.
- **[v1.2] 4. 업적 시스템 (Achievement System)의 검증 모델 및 관리 정책 수립** (F-23)
  - **동작 방식**:
    - 숨겨진 업적(Hidden Achievement) 특성상 조건은 평소에 비공개하며, 달성 시점에 알림 및 주행 상세 로그 화면 내 뱃지 제공.
    - 대상 구간 예시: 유명산 코너링 구간, 느랏재 고개 등 특정 GPS 경위도 중심 및 반경 $R$ (예: $1\text{ km}$) 내.
  - **오인 감지 방지 (False-Positive Prevention) 검증 로직**:
    - 단일 GPS 포인트 진입만으로 판단하지 않고, 진입(Entry) - 경유(Waypoints) - 진출(Exit)로 구성된 **경로 궤적(Trajectory) 정합성**을 검증함.
    - 대상 구간 내에 머무르는 동안 `GpsSignalGuard`가 판단한 `VALID` 좌표가 연속해서 최소 $N$개 이상(예: 30초 이상) 발생해야 함.
    - 속도 제약: 완주 시 평균 속도가 $30\text{ km/h} \sim 80\text{ km/h}$ 사이여야 하며, $100\text{ km/h}$를 초과하는 위험 폭주 주행이나 $5\text{ km/h}$ 이하의 도보/정차 구간은 업적 대상에서 제외함.
    - 안전 요건: 해당 구간 통과 중 급제동 이벤트 단계가 `HARD`인 경우가 1회도 발생하지 않아야 함.

## 7. 비기능적 요구사항 (Quality)

### 7.1 배포 안정성 및 로깅 도구 (Crashlytics 연동)
- **Crashlytics 활성화 정책**: 
  - **Debug 빌드**: 개발 단계에서의 불필요한 로그 수집 및 혼선을 방지하기 위해 Crashlytics 수집 대상에서 제외합니다.
  - **Release 빌드**: 앱 배포 및 운영 단계의 오류 추적을 위해 Crashlytics 수집을 활성화합니다.
- **안정적 배포 환경 구축**:
  - 로컬 개발 환경에서 `google-services.json` 파일이 없을 때 빌드 에러가 나는 것을 방지하기 위해 Gradle 플러그인에 조건부 빌드 로직을 적용합니다.
  - 구글 플레이 릴리즈 배포 시에는 `google-services.json` 키가 누락되지 않도록 빌드 타임 주입 파이프라인(CI/CD)을 구축 및 관리합니다.
- **오류 리포팅 범위**:
  - 앱 크래시(Crash)뿐만 아니라 주요 예외 처리 구간(예: GPS 신호 가드 필터링 오류, 센서 정합성 오류 등)에 대해 비치명적 에러(Non-fatal Event)를 수집하도록 `FirebaseCrashlytics.getInstance().recordException(...)` 코드를 적용합니다.
