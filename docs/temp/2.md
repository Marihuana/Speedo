# [보고서] PRD 작성 및 관리 시스템 분석 결과

**작성일:** 2026년 6월 13일
**대상:** PRD 에이전트 워크플로우 (Android-PRD-Builder, prd-developer)

---

## 1. 개요
본 보고서는 현재 Speedo 프로젝트에서 사용 중인 PRD 작성 및 관리 시스템의 구조를 검토하고, 개발 효율성 및 문서 유지보수 관점에서의 강점과 미흡한 점을 분석하여 개선 방향을 제시합니다.

---

## 2. 주요 분석 결과

### 2.1 주요 강점 (Strengths)
1. **객관적 근거 기반(Evidence-based):** 기존 코드를 역공학하여 `파일:라인` 단위로 근거를 명시함으로써 문서의 신뢰성과 객관성을 확보함.
2. **엄격한 모호성 관리:** 임의 판단을 배제하고 `🔲 TBD` 및 `clarification_required` 프로세스를 통해 사용자의 확답을 체계적으로 수집함.
3. **정밀 업데이트(Surgical Update):** `replace` 도구를 활용하여 기존 문서의 구조를 훼손하지 않고 필요한 부분만 수정하는 협업 전략을 채택함.
4. **아키텍처 정렬:** Jetpack Compose, Hilt, Clean Architecture 등 최신 안드로이드 권장 사양을 PRD 단계부터 구조화함.

### 2.2 미흡한 점 및 리스크 (Weaknesses & Risks)
1. **유지보수 비용의 과다:** 
   - `파일:라인` 형식의 근거 명세는 코드 수정 시 즉시 구식 정보가 되어 문서 신뢰도를 저하시킴.
2. **과도한 구현 상세 명세(Over-specification):** 
   - Hilt 바인딩 방식(@Binds, @Provides)이나 구체적인 리소스 ID를 PRD에 미리 정의함으로써 개발 자율성을 제약하고 문서 수정을 빈번하게 만듦.
3. **도구 활용의 경직성:** 
   - 긴 문서에서 `replace` 도구 사용 시 사소한 포맷(공백 등) 차이로 인해 업데이트 실패 가능성이 존재함.
4. **시스템 안정성 명세 부족:** 
   - 사용자 인터페이스(UI) 위주의 명세에 비해, 안드로이드의 프로세스 종료(Process Death) 대응이나 데이터 무결성 보존 정책 등의 기술적 깊이가 부족함.

---

## 3. 개선 제안 (Recommendations)

### 3.1 명세 방식의 고도화
- **심볼 기반 참조:** `파일:라인` 대신 `파일:클래스/메서드명` 방식으로 근거를 기록하여 코드 변경에 유연하게 대응할 것을 제안합니다.
- **추상화 계층 분리:** 구현 상세(DI 바인딩, 리소스 식별자)는 PRD 본문이 아닌 '기술 부록'으로 분리하거나 정책적 명시로 대체합니다.

### 3.2 문서 구조 및 도구 최적화
- **Open Items 테이블화:** `🔲 TBD` 항목을 문서 하단에 독립된 테이블 형식(`ID | 항목 | 상태 | 질문`)으로 관리하여 에이전트의 자동 업데이트 효율을 높입니다.
- **검증 시나리오 구체화:** PRD의 테스트 전략이 실제 테스트 코드 파일과 매핑되도록 규칙을 강화하여 검증의 완결성을 높입니다.

### 3.3 안드로이드 특화 요구사항 강화
- **생명주기 대응 명시:** `SavedStateHandle` 활용이나 비정상 종료 시의 데이터 복구 정책을 "시스템 이벤트" 섹션에 필수 항목으로 추가합니다.

---

## 4. 결론
현재 시스템은 신뢰도 높은 명세를 위한 탄탄한 기초를 갖추고 있습니다. 향후 **"문서 유지보수의 자동화"**와 **"구현의 유연성"**을 확보하는 방향으로 가이드를 미세 조정한다면, 규모가 큰 프로젝트에서도 지속 가능한 개발 파트너가 될 것으로 기대됩니다.
