# 📄 안드로이드 개발 스킬 검토 보고서

## 1. 검토 개요
* **대상**: `.claude/skills/android-development/` 내 `SKILL.md` 및 `references/*.md`
* **목적**: 프로젝트 아키텍처(MVI/UDF) 및 최신 안드로이드 개발 표준(API 34+) 정합성 검증

## 2. 긍정적인 측면 (Strengths)
* **명확한 계층 분리**: UI, Domain, Data 계층의 역할 분담과 Repository 구현체(`Impl`) 분리 정책이 프로젝트 구조와 일치하게 잘 정의되어 있습니다.
* **MVI/UDF 표준화**: `StateFlow`와 `SharedFlow`를 이용한 상태 및 일회성 이벤트 처리 패턴이 구체적인 코드 예시와 함께 제시되어 있어 에이전트의 오해를 최소화합니다.
* **실전적 가이드**: Speedo 앱의 핵심인 `RecordingService`와 `Foreground Service` 구현 패턴이 상세히 명세되어 있습니다.

## 3. 발견된 문제점 및 개선 필요 사항 (Identified Issues)

### 🔴 문제점 1: 백그라운드 리소스 소모 (Lifecycle Optimization)
* **내용**: `ui_layer.md`에서 Compose 상태 수집 시 `collectAsState()` 사용 지침이 부족함.
* **리스크**: 앱이 백그라운드로 전환되어도 GPS/센서 데이터 수집이 중단되지 않아 불필요한 배터리 소모 및 메모리 낭비가 발생할 수 있음.

### 🔴 문제점 2: 명령형 액션의 Flow 오남용 위험 (Flow vs Suspend)
* **내용**: Repository의 모든 함수를 `Flow`로 노출하도록 유도하는 경향이 있음.
* **리스크**: '기록 시작', '영점 보정' 등 단발성 액션(Command)까지 `Flow`로 설계할 경우 불필요한 보일러플레이트가 발생하고 로직이 복잡해짐.

### 🔴 문제점 3: DI 컴포넌트 범위 설정(Scope) 모호성
* **내용**: Hilt 모듈 예시가 모두 `SingletonComponent`로 고정되어 있음.
* **리스크**: 싱글톤이 불필요한 객체까지 앱 프로세스 전체 수명 주기에 묶여 메모리 효율이 저하됨.

### 🔴 문제점 4: Android 14(API 34) 최신 정책 미반영
* **내용**: `service_layer.md`에 `foregroundServiceType` 선언 및 런타임 명세 지침이 누락됨.
* **리스크**: Android 14 이상 기기에서 주행 기록 시작 시 `MissingForegroundServiceTypeException` 크래시 발생 가능.

---

## 4. 💡 스킬 개선 제안 (Recommendations)

**① UI 수집 표준 고도화 (`ui_layer.md`)**
* `collectAsStateWithLifecycle()` 사용을 기본 원칙으로 명시.

**② 데이터 레이어 함수 정의 규칙 추가 (`data_layer.md`)**
* 관찰 가능한 데이터 스트림은 `Flow`로, 단발성 명령은 `suspend` 함수로 명확히 구분.

**③ Hilt Scope 가이드 보강 (`di_hilt.md`)**
* `@ViewModelScoped`, `@ActivityScoped` 등 생명주기에 맞는 적절한 범위 설정 가이드 추가.

**④ Android 14 대응 지침 추가 (`service_layer.md`)**
* `Manifest` 내 `foregroundServiceType="location"` 선언 및 `startForeground()` 호출 시 타입 명시 필수화.
